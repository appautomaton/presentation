# Bain Process Reference

Updated: 2026-03-03
Companion: [catalog.md](catalog.md) — per-framework intent, inputs, outputs, and decision context.

## Agentic usage contract

When this file is loaded, follow these rules:

1. Apply decision-first logic before analytical depth.
2. Enforce a decision gate for every analysis block.
3. Run all work through the three-loop spiral model.
4. Maintain traceability from decision → option → condition → analysis → activation.
5. Stop analysis when decision confidence threshold is met.

## Operating philosophy

Core orientation:

- Product focus: measurable business result.
- Primary design object: executive decision quality.
- Analytical principle: minimum sufficient analysis.
- Engagement architecture: iterative spiral, not linear stack.
- Execution principle: results tracking with behavioral change control.

Decision gate rule:
- If completion of an analysis block does not change available decisions, redesign or remove that block.
- **Decision confidence threshold**: for reversible decisions, 70% confidence is sufficient to stop analysis. For irreversible decisions (major acquisitions, restructurings), demand 90%+.

### How Bain differs from McKinsey at the root

McKinsey's Problem Statement is open-ended — "How can we improve margin?" — and does not presuppose the answer format. The downstream analysis explores broadly until a hypothesis is proved or killed.

Bain's Decision Brief is closed-form — "Should we acquire Target Y at $2.4B, or build organically, or partner?" — and presupposes the answer is a discrete choice. The downstream analysis has a built-in termination condition: when evidence is sufficient for the CEO to choose confidently among the options, analysis stops. Not before, not after.

This means every Bain analytical workstream exists for one reason: to validate or invalidate a specific condition in a conditional hypothesis. If a workstream can't be traced back to a condition, it shouldn't exist.

## Three-loop spiral model

Loop 1: Strategic Due Diligence
- Defines decision space, option set, and conditional hypotheses.

Loop 2: Value Architecture
- Quantifies economics, response risk, customer value, and feasibility across finalist options.

Loop 3: Activation Design
- Converts selected option into result cards, sprint sequencing, behavior interventions, and delivery governance.

Iteration policy:
- Loops run concurrently and iterate to increasing precision.
- New findings in any loop can trigger updates in other loops.

## Loop 1: Strategic Due Diligence

### L1.1 Decision framing

Primary artifact: Decision Brief

Decision Brief required fields:

1. The Decision (closed-form choice).
2. Decision Criteria (3 to 5 criteria).
3. Decision Timeline (decision deadline and key dates).
4. Decision Stakeholders (named decision actors).

### L1.2 Option generation and constraint mapping

Primary artifact: Option Map

**Options funnel** — generate broadly, then narrow through principled filtering:

1. **Multi-channel input**: gather ideas from staff submissions, data analysis, and interviews simultaneously. Cast wide.
2. **Guiding principles filter**: establish non-negotiable principles before analysis to bound the solution space. Costs evaluated against relative value, not absolute cost. Any option that passes has pre-clearance from stakeholders.
3. **Constraint mapping**: classify surviving options by constraint class:
   - Financial constraints.
   - Organizational constraints.
   - Temporal constraints.

Option state policy:
- Feasible options remain active.
- Infeasible options remain visible and tagged inactive.

### L1.3 Screening criteria application

Primary artifact: Rapid Screen Matrix

Method:
- Rate each option against each criterion using H/M/L.
- Reduce option set to finalists for deep analysis.

### L1.4 Conditional hypothesis articulation

Primary artifact: Conditional Hypothesis Map

Hypothesis form:
- "Option X is preferred if conditions C1..Cn hold."

This is structurally different from McKinsey's assertive hypothesis format ("We believe X is true because of Y"). McKinsey hypotheses assert a belief to be tested. Bain hypotheses define the conditions under which each option wins — then design workstreams to test those conditions directly.

Traceability requirement:
- Each condition is mapped to one named Loop 2 analytical workstream.
- Loop 2's entire work plan is derived from these conditions — every workstream exists solely to validate or invalidate a specific condition.

### Loop 1 quality gates

1. Decision is expressed as discrete choice.
2. Option set is exhaustive enough for executive decision.
3. Constraint tags exist for every option.
4. Finalist set is explicitly justified.
5. Every conditional hypothesis has mapped validation workstreams.

## Loop 2: Value Architecture

Loop 2 uses one unified pipeline for all finalist options.

This is structurally different from McKinsey's Tier 2, which offers three parallel engines (Benchmark, Value Driver Tree, GGD) selected per engagement. Bain runs all finalists through the same five-module pipeline. McKinsey's Tier 2 is diagnostic (decomposing current performance to find gaps). Bain's Loop 2 is projective (building each option's future economic profile to compare relative attractiveness).

### L2.1 Economics modeling

Primary artifact: Bain Value Model

Model blocks:

1. Revenue build (volume, price, mix).
2. Cost structure (variable, fixed, incremental investment).
3. Returns (NPV, IRR, payback, ROIC).

Scenario policy:
- Base case.
- Upside case.
- Downside case.
- **Realization haircut**: apply 60-80% to identified savings in typical environments, 40-60% in regulated/politically constrained environments. Present ranges, not maximums.

### L2.2 Competitive response simulation

Primary artifacts:

1. Competitor Response Profiles.
2. Competitive Response Matrix.

Response profile fields:

1. Strategic posture.
2. Capability to respond.
3. Economic incentive to respond.

Integration rule:
- Expected competitor responses must be reflected in downside economics assumptions.

### L2.3 Customer value proposition testing

Primary artifacts:

1. JTBD segmentation map.
2. Value proposition matrix.
3. NPS impact estimation.

Value proposition fields:

1. Target segment.
2. Key benefit.
3. Quantified value.
4. Proof points.

Linkage rule:
- NPS impact estimates are translated into economics assumptions where required.

### L2.4 Implementation feasibility assessment

Primary artifact: Feasibility Radar

Assessment dimensions:

1. Talent readiness.
2. Technology readiness.
3. Process readiness.
4. Capital readiness.
5. Culture readiness.

Scoring method:
- Structured maturity interviews with calibrated scoring.

### L2.5 Integrated option scorecard

Primary artifact: Integrated Option Scorecard

Scorecard domains:

1. Economics.
2. Competitive robustness.
3. Customer value.
4. Feasibility.

Decision policy:
- Use lexicographic decision logic instead of simple weighted averaging.

Default lexicographic sequence:

1. Economics hurdle screen.
2. Competitive robustness check.
3. Customer value x feasibility ranking.

### Loop 2 quality gates

1. All finalist options use the same pipeline structure.
2. Scenario assumptions are explicit and auditable.
3. Competitive response effects are reflected in option economics.
4. Customer value claims include quantified evidence.
5. Feasibility scoring uses structured protocol.
6. Final recommendation logic is explicitly documented.

## Loop 3: Activation Design

Loop 3 starts in parallel with Loops 1 and 2 and is refined after recommendation lock.

### L3.1 Organizational readiness and decision rights

Primary artifact: RAPID Decision Map

RAPID roles:

1. Recommend.
2. Agree.
3. Perform.
4. Input.
5. Decide.

### L3.2 Results definition

Primary artifact: Result Cards

Result Card required fields:

1. Result description.
2. Result owner.
3. Result metric.
4. Baseline value.
5. Target value.
6. Target date.
7. Measurement frequency.
8. Data source.
9. Leading indicators.

### L3.3 Initiative sequencing

Primary artifact: 90-Day Sprint Plan (Micro-battles)

Bain's execution unit is the micro-battle: a focused, cross-functional team that takes on a single winnable fight within a 90-day cycle. The logic is that large-scale transformation fails when it tries to change everything at once. Instead, select the 3-5 battles where a win is most visible and most achievable, resource them with dedicated teams, execute fast, prove the model works, then scale the approach. Each micro-battle is a self-contained proof point.

Sprint structure:

1. Sprint objective (tied to a specific Result Card).
2. Activities.
3. Sprint owner (single accountable person with authority to act).
4. Success criteria (measurable, binary: won or lost).

Design principle:
- Keep sprint dependencies minimal and autonomy high.
- Sequence battles so early wins fund later ones — both financially and politically.

### L3.4 Behavioral change design

Primary artifact: Behavioral Change Map

Required fields per critical behavior:

1. Current state.
2. Target state.
3. Barrier category.
4. Intervention action.

Barrier categories:

1. Skill.
2. Motivation.
3. Structure.
4. Information.

### L3.5 Results Delivery Office blueprint

Primary artifact: RDO Blueprint

Required elements:

1. Reporting structure.
2. Team composition.
3. Meeting cadence.
4. Escalation protocol.
5. Results Dashboard design.

Dashboard requirement:
- Show both status and trajectory per metric.

### Loop 3 quality gates

1. Decision rights are mapped before full rollout.
2. Every result has owner, metric, target, and leading indicators.
3. Sprint plans are linked to result cards.
4. Behavioral barriers have explicit interventions.
5. Dashboard supports proactive intervention.

## Facts vs Perspectives

Bain's signature decision-quality device. Separates observed data from the firm's interpretation on analytical slides.

**Structure:** Left column (~55%): "KEY FACTS" — quantified findings with source citations. Objective. The data speaks. Right column (~45%): "PERSPECTIVES" — interpretation, implications, recommended action. Subjective. The firm's judgment.

**Why it matters:** Forces intellectual honesty — the reader can evaluate whether the data actually supports the recommendation. Prevents the common failure mode of mixing observation with opinion until neither is trustworthy.

**When to use:** On any analytic slide where the recommended action could be contested. At L2+ density. At L1, the separation collapses into assertion-with-evidence (the action title carries the perspective, the exhibit carries the facts).

**Quality test:** Cover the Perspectives column — do the facts alone suggest a different conclusion? If yes, the Perspectives column is doing real work (good). If the facts only support one conclusion, the split is theatrical (bad — just assert the conclusion).
